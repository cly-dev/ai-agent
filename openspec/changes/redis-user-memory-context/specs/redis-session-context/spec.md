## ADDED Requirements

### 需求:对话上下文必须绑定 sessionId 且与会话生命周期一致
系统必须将对话上下文存储在 Redis 中，且键必须包含平台 `Session.id`（`sessionId`）。系统必须禁止在缺少有效 `sessionId` 时写入会话上下文。

#### 场景:按会话读写上下文
- **当** 调用方提供合法 `sessionId`
- **那么** 系统必须使用与该 `sessionId` 唯一对应的 Redis 键读写上下文

#### 场景:跨会话隔离
- **当** 两个不同 `sessionId` 并发读写
- **那么** 系统必须保证两个会话的 Redis 键互不干扰

### 需求:会话上下文必须支持 TTL 以防止无限增长
系统必须为会话上下文键配置生存时间（TTL）或等价的自动过期机制，并必须在文档中说明默认值与覆盖方式。

#### 场景:会话长期无活动
- **当** 超过配置的 TTL 未对该会话上下文进行续期或更新
- **那么** Redis 中的该会话上下文键必须过期失效

#### 场景:活跃对话续期
- **当** 调用方在 TTL 内更新该会话上下文
- **那么** 系统必须根据配置刷新或延长该键的生存时间

### 需求:会话上下文与数据库消息的关系必须可解释
系统必须在规范或实现文档中说明：Redis 中的会话上下文是否为 PostgreSQL `Message` 行的缓存/派生视图，以及重建策略（例如懒加载回填或仅运行时态）。

#### 场景:Redis 上下文缺失但数据库有历史消息
- **当** Redis 中无该会话上下文而数据库仍存在该 `sessionId` 的消息
- **那么** 系统必须定义是否从数据库重建上下文或返回空上下文，且行为必须一致、可测试
