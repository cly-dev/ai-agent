## ADDED Requirements

### 需求:用户记忆必须绑定 userId 且使用独立 Redis 键空间
系统必须将用户级记忆数据存储在 Redis 中，且键必须包含平台 `User.id`（`userId`），禁止在缺少有效 `userId` 时写入或读取用户记忆。

#### 场景:按用户读写记忆
- **当** 调用方提供合法 `userId` 并请求读取或更新该用户的记忆
- **那么** 系统必须使用与该 `userId` 唯一对应的 Redis 键完成操作，且不得访问其他用户的记忆键

#### 场景:拒绝匿名记忆写入
- **当** 调用方未提供 `userId` 或 `userId` 无效
- **那么** 系统必须拒绝写入并返回错误

### 需求:用户记忆值必须采用可版本化的结构化载荷
系统必须以可解析的结构（例如 JSON）存储用户记忆载荷，并必须支持在载荷内携带可选的版本或更新时间元数据，以便调用方做合并策略。

#### 场景:读取到结构化记忆
- **当** Redis 中存在该用户的记忆键
- **那么** 系统必须返回解析后的结构化数据或等价的强类型对象

#### 场景:键不存在
- **当** Redis 中不存在该用户的记忆键
- **那么** 系统必须返回空记忆或等价空结构，且不得视为系统错误

### 需求:用户记忆必须支持删除与可配置生存时间
系统必须支持按 `userId` 删除用户记忆键；必须支持为记忆键设置 TTL（包括“无 TTL”配置），且 TTL 行为必须在配置中可文档化。

#### 场景:用户注销或合规删除
- **当** 业务触发删除某用户的记忆
- **那么** 系统必须删除对应 Redis 键或使其不可再被读取
