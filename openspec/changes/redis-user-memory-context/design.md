## 上下文

- 平台已有 PostgreSQL 模型：`User`、`Session`、`Message`，分别承载用户、会话与消息持久化。
- Agent 推理链路需要两类数据：**跨会话、可累积的用户记忆**；**单会话内、高频读写的对话工作上下文**。
- 目标代码落点：`src/core/memory`，与现有 `src/core/llm` 并列，供 Agent/对话服务注入使用。

## 目标 / 非目标

**目标：**

- 通过 Redis 提供**用户维度**与**会话维度**的键值抽象，支持设置、读取、追加/合并、删除与可选 TTL。
- 键名必须绑定平台主键（`userId`、`sessionId`），避免跨用户泄漏。
- 在 Redis 不可用时具备明确失败语义（快速失败或降级策略二选一，见决策）。
- 配置可通过环境变量注入，符合 Twelve-Factor。

**非目标：**

- 不替代 `Message` 表的审计与权威历史；不强制在本变更中实现「全量同步 PG ↔ Redis」的复杂 ETL。
- 不规定具体 Prompt 拼装格式（由调用方把 Redis 中读取的结构转为模型输入）。
- 不实现向量检索或长期记忆图谱（若需后续单独提案）。

## 决策

1. **客户端库**：采用 **`ioredis`**（成熟、支持集群/哨兵、与 Nest 生态常见）。替代方案 `redis` v4+ 亦可，但团队统一选 ioredis 以降低连接管理差异。
2. **键命名空间**：统一前缀 `agent:`，例如  
   - 用户记忆：`agent:memory:user:{userId}`  
   - 会话上下文：`agent:context:session:{sessionId}`  
   值格式默认 **JSON 字符串**（UTF-8），便于版本字段与多字段打包。
3. **服务拆分**：  
   - `MemoryModule` 注册全局或特性模块；  
   - `UserMemoryStore` / `SessionContextStore`（或单一 `MemoryService` 内分方法）对外暴露类型化 API。  
4. **TTL**：用户记忆键支持可配置 TTL（默认较长或 0 表示不过期）；会话上下文键**必须**支持 TTL（滑动续期），防止僵尸键占满内存。具体默认值在实现任务中写死为常量并可 env 覆盖。
5. **与 PG 的一致性**：首次读取会话上下文时**允许**从 `Message` 表懒加载回填 Redis（可选实现）；写入 Redis **不自动**写回 PG（消息仍以现有 API/服务为准）。
6. **不可用行为**：默认 **连接失败在启动期记录错误**；运行期单次操作失败向调用方抛业务异常或返回 Result，由 Agent 层决定是否跳过记忆（不在本设计强制）。

## 风险 / 权衡

| 风险 | 缓解 |
|------|------|
| Redis 宕机导致 Agent 质量下降 | 调用方可降级为空记忆；健康检查与告警；关键路径可选同步读 PG |
| 键空间膨胀 | 会话上下文强制 TTL；用户记忆定期清理任务（后续） |
| 多实例部署下并发写丢失 | 对同一 key 使用 Redis 事务或 Lua 脚本做合并；或接受 last-write-wins 并在 spec 中声明 |
| 敏感数据驻留内存 | 传输 TLS；Redis ACL；键内不存明文密码；遵守数据保留策略 |

## 迁移计划

1. 部署 Redis 实例，配置 `REDIS_URL`（或 HOST/PORT/PASSWORD）。
2. 发布应用版本：引入 `MemoryModule`，默认不改变现有 HTTP API。
3. 灰度：先在内部环境验证连接与键读写，再打开 Agent 路径上的记忆读取。
4. 回滚：关闭功能开关或移除模块导入；Redis 中数据可保留或 `FLUSHDB`（仅非共享库时）。

## 待定问题

- 用户记忆是否需要**多版本/时间旅行**（仅最新 vs 历史列表）？
- 会话上下文窗口是按**条数**截断还是按 **token 预算**截断（若在服务端做）？
- 是否与现有 **SSE / Chat** 模块联动触发上下文更新（事件驱动）？
