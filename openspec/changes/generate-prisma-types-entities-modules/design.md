## 上下文

当前项目已经完成 Nest + Prisma 初始化，并在 `prisma/schema.prisma` 中定义了用户、会话、消息、工具、技能、Agent 及其关联模型。团队目标是让数据库模型变化后可快速同步到应用层类型与模块结构，避免手工创建实体类和模块时出现命名不一致、可空性遗漏、关系字段遗漏等问题。

生成范围聚焦在应用层静态代码：

- TypeScript 类型定义（字段类型、可空性、关系引用）
- 实体类（用于接口返回或领域对象表达）
- Nest 模块骨架（module/service/controller 或最小模块结构）

## 目标 / 非目标

**目标：**

- 提供一个可重复执行（幂等）的代码生成流程，以 `prisma/schema.prisma` 为单一事实源。
- 生成物目录结构和命名统一，便于后续手写扩展与代码审查。
- 生成规则覆盖标量字段、可选字段、时间字段、JSON 字段和基本关系字段。
- 支持按模型增量生成和全量重建两种模式。

**非目标：**

- 不在本次设计中替代 Prisma Client 本身的数据访问能力。
- 不自动生成复杂业务逻辑（鉴权、校验策略、事务编排）。
- 不处理跨服务代码分发，仅关注当前仓库内模块代码。

## 决策

### 决策 1：使用 schema 解析 + 模板渲染方式生成代码

- 选择：以 Prisma schema AST/元数据为输入，通过模板渲染输出 `types`、`entities`、`modules`。
- 原因：相比运行时反射，构建期生成稳定、可审查、易做差异管理。
- 备选方案：
  - 直接依赖 Prisma 生成的 client 类型：无法满足 Nest 模块和实体类文件结构需求。
  - 手工维护实体与模块：维护成本高、容易与 schema 漂移。

### 决策 2：生成目录采用“可生成层 + 可手写层”隔离

- 选择：将生成文件放到固定目录（如 `src/generated/**`），业务手写扩展放到非 generated 目录。
- 原因：避免二次生成覆盖手工代码；升级模型时只需重跑生成。
- 备选方案：
  - 直接覆盖业务模块目录：风险高，容易误覆盖手工代码。

### 决策 3：关系字段默认生成引用类型而非深层嵌套

- 选择：在实体类型中优先生成轻量关系引用（id 或简化对象），通过配置启用深层嵌套。
- 原因：控制生成复杂度和循环依赖，避免类型爆炸。
- 备选方案：
  - 默认深层嵌套：在模型较多时类型体积和编译开销显著增加。

### 决策 4：命令行提供 dry-run 与 overwrite 开关

- 选择：生成命令支持预览变更和覆盖策略。
- 原因：提高可控性，降低误操作风险。
- 备选方案：
  - 无开关直接写盘：开发体验较差，不利于审核。

## 风险 / 权衡

- [模型关系复杂导致生成规则不完整] → 先覆盖 80% 常见关系（1-1/1-N/N-N），复杂场景允许手工扩展并记录 TODO。
- [生成目录与手写目录边界不清] → 在文档和目录命名上强制约定 `generated` 前缀并加注释说明“请勿手改”。
- [二次执行差异不可预测] → 统一排序规则、稳定模板输出，配合 dry-run 先预览。
- [后续 schema 演进导致模板老化] → 将字段映射规则集中在单模块，新增模型特性时只改映射层。

## Migration Plan

1. 引入代码生成脚本与模板目录。
2. 按当前 schema 首次全量生成类型/实体/模块。
3. 对生成结果做一次人工校验并提交基线。
4. 在团队流程中约定：schema 变更后必须先执行生成命令再提交。
5. 若生成规则异常，允许回滚到上一版本生成器并保留旧产物。
