## 上下文

当前系统将业务用户与后台管理权限统一建模在 `User` + `Role` 中。该建模方式无法区分两类权限边界：  
1) 系统管理权限（配置 Tool/Skill/Integration）；  
2) 业务能力权限（用户可调用哪些 Tool）。  
此次重构涉及 `prisma/schema.prisma`、实体类型定义与相关模块边界，属于跨模块数据结构调整。

## 目标 / 非目标

**目标：**
- 将用户体系拆分为 `User`（业务用户）和 `AdminUser`（后台管理员）两个独立模型。
- 建立后台管理员角色枚举 `AdminRole`：`SUPER_ADMIN`、`OPERATOR`、`VIEWER`，并定义权限边界。
- 在业务用户侧建立 `UserType` + `UserRole` 到 Tool 子集的授权关系，明确“能用哪些能力”。
- 保持数据结构简单，避免一次性引入复杂 RBAC 结构，但为后续升级保留扩展位。

**非目标：**
- 不在本次变更中引入完整多层级 RBAC（如资源级策略表达式、策略引擎）。
- 不在本次变更中改造对话/下单等业务流程逻辑，仅调整可支持的权限与身份数据结构。
- 不在本次变更中调整与本主题无关的历史模型字段。

## 决策

### 决策 1：拆分 `User` 与 `AdminUser` 主体
- 选择：新增独立 `AdminUser` 模型，不再复用 `User` 作为后台登录主体。
- 原因：从结构上隔离后台权限与业务权限，避免业务侧字段影响后台鉴权模型。
- 备选方案：
  - 在 `User` 中增加 `isAdmin` 与 `adminRole` 字段：实现快，但长期会持续耦合两套权限语义，放弃。

### 决策 2：后台权限采用固定枚举而非动态角色表
- 选择：使用 `AdminRole` 枚举（`SUPER_ADMIN`、`OPERATOR`、`VIEWER`）。
- 原因：当前后台权限边界清晰且数量稳定，枚举足够表达且迁移成本最低。
- 备选方案：
  - 新建 `AdminRole`/`AdminPermission` 关系表：更灵活，但超出“结构简单”目标，暂不采用。

### 决策 3：业务侧权限采用“身份分类 + Tool 子集授权”
- 选择：在 `User` 中定义身份枚举（`UserType`）与业务角色枚举（`UserRole`），通过映射关系限制可用 Tool 子集。
- 原因：直接满足“控制用户能用哪些能力”的目标，且便于未来替换为更细粒度 RBAC。
- 备选方案：
  - 继续依赖当前 `RoleSkill` 间接控制 Tool：抽象层次不贴合需求，业务表达不直观，放弃。

### 决策 4：迁移期间采用双轨兼容窗口
- 选择：在迁移期保留旧字段读取能力，通过脚本回填并逐步切流。
- 原因：降低生产环境切换风险，便于回滚。
- 备选方案：
  - 一次性强切：实施简单但风险高，不符合安全迁移要求。

## 风险 / 权衡

- 旧数据语义映射风险（旧 `Role` 到新 `UserType/UserRole`）  
  → 通过明确映射表 + 迁移前核验脚本缓解。
- 管理后台权限切换风险（管理员账户授权缺失）  
  → 预置 `SUPER_ADMIN` 种子账户并在发布前校验关键操作权限。
- 短期内权限模型并存导致代码分支增多  
  → 限定兼容窗口，并在任务中明确下线旧模型时间点。
- 未来 RBAC 升级需要重构映射方式  
  → 当前结构预留枚举与关联扩展位，避免再次拆主体。

## 迁移计划

1. 新增数据结构：`AdminUser`、`AdminRole`、`UserType/UserRole` 与业务用户到 Tool 的授权关系。
2. 编写迁移脚本：将旧 `User`/`Role` 数据映射到新结构，生成默认管理员与权限映射。
3. 应用层兼容：在鉴权层同时支持旧模型读取与新模型读取（短期）。
4. 验证与切换：完成权限回归后切换为新模型优先。
5. 下线旧结构：确认无依赖后移除旧 `Role` 相关路径。

**回滚策略：**
- 保留迁移前数据库快照；
- 在兼容窗口内可切回旧鉴权逻辑；
- 回滚后重新执行增量同步脚本保证数据一致。

- 业务用户到 Tool 子集授权是直接用户绑定、按 `UserRole` 继承，还是两者叠加？
- 是否需要为 `AdminUser` 增加操作审计字段（最后登录、最后操作时间）作为本次基础能力？
